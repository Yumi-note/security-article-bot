### ゼロトラストネットワークアーキテクチャの設計原則：境界防御から「常に検証」への転換  

---

## はじめに（約200文字）  
現代のクラウド化・リモートワーク普及で、従来の「境界防御モデル」は限界です。内部ネットワークを盲信すると、**内部脅威**や**横移動攻撃**による大規模データ漏洩リスク（例：2023年国内企業のインシデント増加）が発生します。ゼロトラスト（Zero Trust）は「何も信頼せず、常に検証する」設計思想で、これを知らないと「防御の盲点」を悪用され、情報漏洩やコンプライアンス違反につながります。本記事では、設計原則を基礎から解説し、支援士試験対策も網羅します。  

⚠️ **試験ではここが出る！**：クラウド移行や働き方改革でゼロトラストが注目される背景を問う問題が頻出。  

---

## 前提知識（約500文字）  
**ゼロトラスト（Zero Trust）**とは、「ネットワーク内部も安全ではない」と仮定し、全てのアクセスを明示的に検証するセキュリティモデルです。対照的な従来モデルを比較しましょう。  

- **従来モデル（境界防御）**：ファイアウォールで「信頼ゾーン（Trust Zone）」を作り、内部通信は無検証。  
- **ゼロトラスト**：信頼ゾーンを廃止し、**マイクロセグメンテーション（Micro-segmentation）**でネットワークを細分化。  

**関連用語**：  
- **SDP（Software-Defined Perimeter）**：ユーザーとリソース間の論理的な境界を動的に構築（RFC 9232）。  
- **ID管理（Identity Management）**：ユーザー認証の基盤（例：SAML, OAuth 2.0）。  

ASCII図：従来モデル vs. ゼロトラスト  
```
[従来モデル]  
インターネット ── [FW] ── 信頼ゾーン（内部全通信許可）  
                          ├─ PC  
                          └─ DB  

[ゼロトラスト]  
インターネット ── [ゲートウェイ] ── マイクロセグメント  
                          ├─ PC（認証必須）  
                          └─ DB（別セグメント・追加認証）  
```  
*解説：信頼ゾーンが消え、各リソースが独立したセグメントに。*

## 基本概念（約600文字）  
ゼロトラストの核心は、**NIST SP 800-207**で定義された3つの設計原則です。  

1. **全てのトラフィックを検証**：通信元（ユーザー/デバイス）・通信先（リソース）を問わず、暗号化と認証を必須化。  
2. **最小権限アクセス**：必要な権限のみを付与（例：DB閲覧可だが更新不可）。  
3. **明示的な認証**：多要素認証（MFA: Multi-Factor Authentication）で継続的検証。  

**たとえ話**：城塞都市（境界防御）から、街中に検問所（ゼロトラスト）を設置する社会へ転換。住民IDを常に確認し、目的地ごとに許可証が必要。  

ASCII図：ゼロトラストのアクセスフロー  
```
ユーザー  
  │  
  ↓ MFA認証（例：パスワード＋生体認証）  
[ポリシーエンジン] ←→ [ID管理システム]  
  │ 許可/拒否を判定  
  ↓  
リソース（例：クラウドDB）  
```  
*解説：アクセス前にポリシーエンジンがID・デバイス状態をリアルタイム評価。*  

⚠️ **試験ではここが出る！**：NIST SP 800-207の原則（特に「最小権限」）が試験の正誤問題で頻出。

## 技術的な深堀り（約700文字）  
具体的な実装では、**ポリシーエンジン**と**プロトコル**が鍵です。  

### 技術要素  
- **多要素認証（MFA）**：NIST SP 800-63Bに準拠。認証因子（知識/所持/生体）を2つ以上組み合わせ。  
- **SDPアーキテクチャ**：コントローラーがアクセス制御（RFC 9232）。  
- **ゼロトラストネットワークアクセス（ZTNA）**：エンドポイントとリソース間の暗号化トンネルを動的構築。  

**認証シーケンス例**（NIST SP 800-207に基づく）：  
1. ユーザーがリソースへアクセス要求。  
2. ポリシーエンジンがIDプロバイダー（例：Azure AD）と連携し、MFAを実行。  
3. デバイス正常性（パッチ適用済み等）をチェック。  
4. アクセス許可時のみ、一時的な暗号化パスを生成。  

ASCII図：SDPを用いたシーケンス  
```
ユーザー──アクセス要求──→ [SDPコントローラー]  
                             │ 1. MFA要求  
                             ↓  
                     [IDプロバイダー]  
                             │ 2. 認証結果  
                             ↓  
ユーザー←─一時トークン発行─ [SDPコントローラー]  
                             │ 3. ポリシー適用  
                             ↓  
ユーザー──暗号化通信──→ [リソース]  
```  
*解説：リソースへの直接アクセスは禁止。コントローラー経由で動的に経路制御。*  

**規格番号**：  
- NIST SP 800-207（ゼロトラストアーキテクチャ）  
- RFC 9232（SDPの基本フレームワーク）

## 攻撃・脆弱性の観点（約500文字）  
ゼロトラスト未導入環境では、**内部脅威**や**横移動攻撃**が容易です。  

### 攻撃フロー例（従来モデルの脆弱性悪用）  
1. フィッシングで内部ユーザーの資格情報を窃取。  
2. 信頼ゾーン内で横移動し、重要DBに無制限アクセス。  
**事例**：CVE-2021-34527（PrintNightmare）— Windowsの印刷機能悪用で権限昇格。内部ネットワークで拡散。  

ASCII図：横移動攻撃の流れ  
```
攻撃者  
  │ 1. 端末Aを侵害（フィッシング）  
  ↓  
端末A ── 無検証通信 ──→ 端末B（同一信頼ゾーン）  
                  │ 2. 権限昇格  
                  ↓  
                DB（データ窃取）  
```  

### ゼロトラスト環境でのリスク軽減  
- マイクロセグメンテーションにより、端末Aから端末Bへの通信が遮断。  
- MFA未実施時はDBアクセス不可。  

⚠️ **試験ではここが出る！**：CVE番号付き脆弱性と攻撃手法の組み合わせ問題が頻出。

## 対策・ベストプラクティス（約500文字）  
実装時のベストプラクティスを箇条書きで整理。  

1. **ID管理の強化**：  
   - MFAを全ユーザーに適用（例：Azure MFA）。  
   - 最小権限ポリシー（例：AWS IAMロールで権限限定）。  

2. **ネットワークの細分化**：  
   - マイクロセグメンテーションでリソースを隔離（例：VMware NSX）。  

3. **継続的監視**：  
   - デバイス正常性を常時チェック（例：EDRソリューション連携）。  

**設定例**（クラウド環境）：  
```yaml
# AWSポリシー例（S3バケットへの最小権限）  
{  
  "Version": "2012-10-17",  
  "Statement": [{  
    "Effect": "Allow",  
    "Action": "s3:GetObject",  
    "Resource": "arn:aws:s3:::example-bucket/*"  
  }]  
}  
```  

ASCII図：ゼロトラスト対策フロー  
```
[ユーザー]  
  │  
  ↓ MFA認証  
[ポリシーエンジン]─アクセス拒否─→ [異常時]  
  │  
  ↓ アクセス許可  
[監視システム]←─ログ連携─ [リソース]  
```  
*解説：認証後も通信を監視し、異常行動を検知。*

支援士試験で頻出のポイントを表形式で整理。頻出度は★の数（最大3つ）。  

| テーマ               | 頻出ポイント                     | 間違えやすい点                     |

| 設計原則（★3）       | NIST SP 800-207の3原則           | 「暗号化は必須」と誤解（認証が核心）|  
| 関連技術（★2）       | SDP（RFC 9232）・ZTNA            | SDPとVPNを混同（VPNは境界防御の技術） |  
| 攻撃対策（★2）       | 横移動攻撃への対応               | 「内部ネットワークは安全」という誤った前提 |

**引っかけ問題パターン**：  
- 「ゼロトラストではファイアウォールが不要」→ **誤り**（FWは内部セグメント間で依然活用）。  
- 「MFAはオプション」→ **誤り**（NIST SP 800-207で継続的認証が必須）。

## まとめ（約200文字）  
ゼロトラストは「常に検証」が核心で、NIST SP 800-207の原則（全検証・最小権限・明示的認証）が基盤です。境界防御モデルでは防げない内部脅威に対応し、マイクロセグメンテーションやSDPで実装します。支援士試験では、原則と攻撃事例（CVE番号付き）をセットで暗記！ 次は、ゼロトラストの実装例（例：Azure Zero Trust）や、関連テーマ「クラウドセキュリティ」を学び、体系的な知識を完成させましょう。  

> **要約**：  
> 1. 従来モデルは内部リスクに脆弱 → ゼロトラストで解決。  
> 2. 設計原則はNIST SP 800-207が基準。  
> 3. 試験対策は「原則＋攻撃事例」の組み合わせを重点的に。
