## ランサムウェアの動作原理と感染経路・復旧手順：防御から試験対策まで徹底解説

## はじめに（約200文字）
ランサムウェアは企業に平均2億円超の被害（警察庁2023年統計）をもたらす深刻な脅威です。知らないと**業務停止・巨額の損害・信用失墜**リスクがあります。例えば、病院で電子カルテが暗号化されると生命に関わる事態に発展。本記事では「なぜ暗号化が解けないのか」「どう感染するのか」を技術背景から解説し、支援士試験頻出ポイントも押さえます。セキュリティエンジニアとして必須の知識です！

## 前提知識（約500文字）
### キーワード解説
- **マルウェア（Malware）**: 悪意あるソフトウェア総称。ランサムウェアはその一種。
- **暗号化（Encryption）**: データを数学的アルゴリズムで変換し読めなくする技術。復元には**鍵（Key）**が必要。
- **C2サーバー（Command and Control Server）**: 攻撃者がマルウェアを遠隔操作するサーバー。
- **非対称暗号（Asymmetric Cryptography）**: 暗号化鍵と復号鍵が異なる方式（例：RSA）。ランサムウェアで多用。

### ネットワーク基本構造（ASCII図）
```
[ユーザーPC] ← HTTPS → [インターネット]
    |                         ↑
    ↓ (感染)                (C2通信)
[ファイルサーバー]       [攻撃者のC2サーバー]
```
⚠️ **試験頻出**: C2通信は「TLS（RFC 8446）」で暗号化され検知が困難。

## 基本概念（約600文字）
ランサムウェアは「**暗号化による人質化→身代金要求**」が核心です。典型的な動作フロー：
1. 感染端末のファイルスキャン
2. 暗号化アルゴリズム（例：AES-256）でファイル変換
3. 復号鍵をC2サーバーへ送信
4. 身代金要求画面を表示

### 動作フロー図（ASCII）
```
1. 配布（メール添付）→ 2. 実行 → 3. 権限昇格（Privilege Escalation）
                          ↓
4. ファイル暗号化(AES)  5. 鍵送信(RSA) → [C2サーバー]
                          ↓
6. 身代金要求表示（「支払わないと鍵削除」）
```
**たとえ話**: 南京錠（AES）で金庫を閉め、鍵（RSA鍵）を遠くに投げるようなもの。南京錠は頑丈だが鍵がなければ開けられない。

## 技術的な深堀り（約700文字）
### 暗号化プロセス詳細
1. **鍵生成**:  
   - ファイル暗号化用の**セッション鍵**（AES-256）を生成。
   - セッション鍵を**RSA-OAEP（RFC 8017）**で暗号化（攻撃者の公開鍵使用）。
   ```python
   # 疑似コード例
   session_key = generate_aes_key()  # ランダムな256bit鍵
   encrypted_key = rsa_encrypt(session_key, attacker_public_key)
   ```

2. **ファイル暗号化**:  
   - 各ファイルをAESで暗号化（モード：CBC/CTR）。NIST SP 800-38A準拠。
   - 拡張子を変更（例：.docx → .locked）。

3. **C2通信**:  
   - 暗号化鍵をTLS 1.3（RFC 8446）でC2サーバーへ送信。
   - ドメイン生成アルゴリズム（DGA）で検知回避。

### シーケンス図（ASCII）
```
[マルウェア]           [C2サーバー]
    | -- TLS Handshake (RFC 8446) → |
    | ← Session Ticket ------ |
    | -- encrypted_key (RSA) → |
    | ← 指令（削除確認）-- |
```
⚠️ **試験対策**: RSA鍵長は2048bit以上が必須（NIST SP 800-57推奨）。試験で「1024bitは脆弱」と出題！

## 攻撃・脆弱性の観点（約500文字）
### 代表的な感染経路
- **フィッシングメール**: 偽装メールの添付ファイル/リンク経由（全感染の70%以上）。
- **脆弱性悪用**:  
  - **EternalBlue（CVE-2017-0144）**: SMBv1の脆弱性。WannaCryで悪用。
  - **Log4Shell（CVE-2021-44228）**: Javaライブラリ脆弱性。

### 攻撃フロー図（ASCII）
```
1. 配布: [フィッシングメール] → 2. 実行 → [ユーザーPC]
3. 拡散: [SMB脆弱性悪用] → 4. 暗号化 → [ファイルサーバー]
                ↓
        [攻撃者にBTC送金要求]
```
**事例**: LockBit 3.0（2023年）は「二重脅迫」を採用。暗号化＋データ窃取を組み合わせ、支払わないとデータ公開。

## 対策・ベストプラクティス（約500文字）
### 予防策（3層防御）
1. **入口対策**:  
   - メールゲートウェイで添付ファイル検査（拡張子偽装検知）。
   - 脆弱性管理（CVEスキャン頻実施）。

2. **拡散防止**:  
   ```mermaid
   graph LR
   A[端末分離] --> B[ネットワークセグメンテーション]
   B --> C[最小権限原則]
   ```

3. **被害軽減**:  
   - **3-2-1ルールのバックアップ**: 3世代のコピーを2種の媒体に1つはオフラインで保存（NIST SP 800-34推奨）。
   - 復旧訓練の実施。

### 復旧手順
1. 感染端末をネットワークから切断（⚠️ **絶対に電源OFFしない**：メモリ内証拠消失）
2. バックアップから復元（暗号化前の世代を使用）
3. インシデント分析（ファイアウォールログ確認）

## まとめ（約200文字）
ランサムウェアは「非対称暗号化＋C2通信」で動作し、主にフィッシングや脆弱性悪用で感染します。対策の核心は**多層防御とオフラインバックアップ**。試験では暗号アルゴリズム（AES/RSA）や感染経路（CVE番号付き）が頻出です。次に学ぶべきは「EDR（Endpoint Detection and Response）」や「インシデント対応フロー」（NIST SP 800-61）で、防御から検知・対応までを体系化しましょう！
