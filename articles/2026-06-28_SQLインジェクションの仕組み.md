## SQLインジェクションの仕組みと完全な対策  
**〜安全なWebアプリ構築の核心を支援士試験対策と共に徹底解説〜**

## はじめに
SQLインジェクションは**20年以上経っても未だに上位を占める致命的な脆弱性**です。IPAの「ソフトウェア等の脆弱性関連情報に関する届出状況」では、2023年でも深刻度「高」の事例が多数報告されています。この脆弱性を放置すると、顧客データの漏洩（例：個人情報3億件流出事件）やウェブサイト改ざん、さらにはサーバー完全乗っ取りに発展します。支援士試験では毎年出題され、実務でも即時対応が求められる必須知識です。

## 前提知識
### Webアプリの基本構造
クライアント（ブラウザ）とWebサーバー（Apache/Nginx）、アプリケーションサーバー（PHP/Python）、データベース（DB）が連携します。

```ascii
[ユーザー] 
    | (HTTPリクエスト GET/POST)
    v
[Webサーバー] 
    | (リクエスト処理)
    v
[アプリケーション] 
    | (SQL生成)
    v
[データベース] <--> [結果を返す]
```

**SQL（Structured Query Language）**: データベース操作言語。例:  
```sql
SELECT * FROM users WHERE id = '入力値' AND password = 'hash値';
```

**動的SQL**: ユーザー入力をもとにSQL文を**文字列連結**で生成する方式。ここに脆弱性が発生します。

## 基本概念
### SQLインジェクションの本質
**「ユーザー入力がSQL文の構造を破壊する」**現象です。悪意ある入力を組み込むことで、意図しないSQLを実行させます。

```ascii
# 正常な例（ユーザー入力: 'Alice'）:
SELECT * FROM users WHERE name = 'Alice'

# インジェクション例（入力: ' OR '1'='1）:
SELECT * FROM users WHERE name = '' OR '1'='1'
```

### 発生プロセス
```ascii
1. ユーザー入力: [悪意ある文字列] → フォーム送信
2. アプリケーション: "SELECT ... WHERE id = '" + 入力 + "'" を生成
3. DBが改ざんされたSQLを実行 → 全データ漏洩
```

**⚠️試験頻出**: 文字列連結（`+`演算子）を使ったSQL生成が根本原因

## 技術的な深堀り
### 攻撃技術の詳細
#### 1. **UNION攻撃**
別テーブルへのアクセスを可能にする:

```sql
' UNION SELECT credit_card_no FROM payments --
```
→ `--`はコメントアウト（DBにより`#`も）

#### 2. **ブラインドSQLi**
エラーや遅延を観測して情報を推測:
```sql
' AND (SELECT SLEEP(5) FROM users WHERE id='admin')--
```

#### 3. **エラーベース攻撃**
DBエラーメッセージから情報収集

### 防御の核心技術
**プレースホルダ（Prepared Statement）**  
SQL文の**構造**と**データ**を分離する仕組み（RFC 3875で定義）:

```java
// 脆弱な例（文字列連結）
String sql = "SELECT * FROM users WHERE id = '" + input + "'";

// 安全な例（Prepared Statement）
PreparedStatement stmt = conn.prepareStatement(
    "SELECT * FROM users WHERE id = ?"  // ? がプレースホルダ
);
stmt.setString(1, input); // 入力値をバインド
```

**バインド機構の動作**:  
```ascii
[SQLテンプレート] 
    SELECT * FROM users WHERE id = ?
        ↓
[入力値] → [エスケープ処理] → パラメータとして"バインド"
        ↓
DBエンジンが構造とデータを別々に解釈 → インジェクション不可能
```

**NIST SP 800-123**（サーバセキュリティガイド）:  
> "Prepared statements with parameterized queries are the most secure way."

## 攻撃・脆弱性の観点
### 実際の攻撃フロー
```ascii
[攻撃者] 
    | 1. 入力フィールドに攻撃コード注入
    v
[アプリ] 
    | 2. 無防備にSQLを組み立て
    v
[DBサーバ] 
    | 3. 管理者権限で実行されるケースも
    v
[結果] → データ窃取 / バックドア設置 / ファイル読み出し
```

**有名事例**:  
- **CVE-2019-19781** (Citrix ADC): SQLi経由で任意コード実行
- **CVE-2020-5405** (Spring Framework): 特定条件下でSQLi可能

**⚠️盲点**: 管理画面やログ検索機能など「外部公開されない機能」も標的に

## 対策・ベストプラクティス
### 多層防御の実装

```markdown
1. **入力値検証** (Input Validation)
   - 許可リスト方式（例: 数字のみ許可）を採用
   - ブラックリスト禁止（`'`のエスケープだけでは不十分）

2. **プレースホルダの必須化**
   - 全てのSQLで静的プレースホルダを使用
   - ORM（Object-Relational Mapping）でも「生SQL」を避ける

3. **最小権限原則**
   - DBユーザー権限を`SELECT`のみに制限
   - `DROP TABLE`等の権限を剥奪

4. **エラーメッセージ制御**
   - 詳細エラーをユーザーに表示しない（RFC 3875 §6.2.5）

5. **WAF（Web Application Firewall）**
   - 緊急措置として有効（根本対策ではない）
```

## 📝 試験対策ポイント

| 論点 | 正答例 | 誤答例（引っかけ） |
|------|--------|-------------------|
| **根本的対策** | 静的プレースホルダ | 動的プレースホルダ |
| **エスケープ有効性** | DB毎に方式が異なる | 万能なエスケープ関数がある |
| **影響範囲** | 認証回避・情報漏洩 | ディスク破壊のみ |
| **検出手法** | DAST＋手動テスト | ソースコード監査のみ |
| **関連法規** | 個人情報保護法 | 電気通信事業法のみ |

**頻出引っかけ**:  
- 「エスケープ処理をすれば安全」→ DB種別依存（MySQLの`\`とPostgreSQLの`''`）
- 「WAF導入で対策完了」→ あくまで補助的手段

## まとめ
SQLインジェクション対策の核心は **「SQL文とデータの分離」** です。プレースホルダの正しい実装に加え、入力検証・最小権限・エラー制御を組み合わせた**多層防御**が必須です。支援士試験では「文字列連結の危険性」「プレースホルダの内部動作」が頻出です。次に学ぶべきは「同一生成元ポリシー（SOP）」や「クロスサイトスクリプティング（XSS）」で、これらを組み合わせた対策が真のセキュアコーディングを実現します。

> **"Security is not a product, but a process." - Bruce Schneier**  
> 対策は一度きりではなく、継続的な確認と改善が必要です。
