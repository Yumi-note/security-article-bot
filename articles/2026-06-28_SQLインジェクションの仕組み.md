## 前提知識（500文字）  
### Webアプリの3層構造  
SQLインジェクションを理解するには、Webアプリケーションの基本構造が不可欠です。  

```
ユーザー(ブラウザ)  
  │  
  │ HTTPリクエスト (例: GET /login?user=alice)  
  ▼  
Webサーバー (Apache/Nginx)  
  │  
  │ SQLクエリ生成 (例: SELECT * FROM users WHERE name='alice')  
  ▼  
データベース (MySQL/PostgreSQL) ← 攻撃対象!
```

- **SQL（Structured Query Language）**：データベース操作言語。`SELECT`, `INSERT`, `DELETE`が主要命令  
- **動的クエリ**：ユーザー入力をもとにSQLを組み立てる処理（例：PHPの`"SELECT * FROM users WHERE id='$_GET[id]'"`）  
- **RFC 3875（CGI仕様）**：HTTPリクエスト経由で入力が渡される仕組みの基盤

## 基本概念（600文字）  
### 「注入」の核心：文字列連結の脆弱性  
SQLインジェクションは、**ユーザー入力がSQL文の一部として解釈される**ことで発生。具体例で見てみましょう。  

**正常なログイン処理**:  
```sql
入力: user = "alice"  
生成SQL: SELECT * FROM users WHERE name='alice'
```

**攻撃時の挙動**:  
```sql
入力: user = "' OR 1=1 -- "  
生成SQL: SELECT * FROM users WHERE name='' OR 1=1 -- '
```
→ `1=1`（常に真）とコメントアウト(`--`)で認証回避  

```
攻撃者          Webサーバー          DBサーバー
  │                  │                    │
  │── user="alice"──→│                    │
  │                  │─ 正当クエリ実行 ──→│
  │                  │←── 認証成功 ──────│
  │                  │                    │
  │─ user="' OR 1=1"→│                    │
  │                  │─ 悪意クエリ実行 ──→│
  │                  │←── 全ユーザー漏洩 ─│
  │←─ 管理者権限取得 ─│                    │
```

> **たとえ話**  
> 「建築図面に『ここに爆弾を設置』と書き加える」行為に似ています。設計図（SQL構文）と材料（入力データ）の区別がないため起きるのです。

## 技術的な深堀り（700文字）  
### 攻撃技術の体系と内部挙動  
#### 1. **Union-based攻撃**  
`UNION`句で別テーブルを強制結合：  
```sql
入力: ' UNION SELECT credit_card FROM payments -- 
生成SQL: SELECT name FROM users WHERE id='' 
         UNION SELECT credit_card FROM payments -- '
```

#### 2. **Blind SQLi**  
エラー/遅延を観測して情報推測：  
```sql
入力: ' AND IF(SUBSTR(database(),1,1)='a', SLEEP(5), NULL) -- 
→ 5秒遅延ならDB名の先頭が'a'
```

#### 3. **エラー型攻撃**  
意図的なエラーメッセージから情報取得：  
```sql
入力: ' AND GTID_SUBSET(@@version,0) -- 
→ MySQLバージョンがエラーに出力される
```

**パケットレベルでの悪用**:  
```
HTTPリクエスト（実際の例）:
GET /product.php?id=1%27%20UNION%20SELECT%20NULL,@@version,NULL-- 

デコード後: id=1' UNION SELECT NULL,@@version,NULL--
```

**規格との関係**:  
- **NIST SP 800-123**: 入力検証のガイドライン  
- **RFC 7578 (セキュアコーディング)**: データとコードの分離原則  
- **CWE-89**: SQLインジェクションの脆弱性ID  

⚠️ **試験ではここが出る！**  
`UNION`攻撃で必要な「カラム数調整」手法（`ORDER BY`による試行）は頻出。

## 攻撃・脆弱性の観点（500文字）  
### 現実世界の被害事例  
#### 1. **CVE-2019-0211（Apache脆弱性）**  
`mod_session_sql`モジュールでのSQLi。認証回避により管理者権限奪取。  

#### 2. **CVE-2017-5638（Apache Struts）**  
OGNL式インジェクションから派生したSQLiで、Equifaxの1.4億人データ流出。  

#### 攻撃フローの全体像  
```
攻撃者  
  │  
  │ 1. 入力欄に悪意のペイロード送信  
  ▼  
Webアプリ (脆弱な動的クエリ生成)  
  │  
  │ 2. 未検査の入力がSQLに連結  
  ▼  
データベース  
  │  
  │ 3. 機密データを攻撃者へ返却  
  ▼  
攻撃者 (データ搾取/改ざん完了)
```

**被害の連鎖**:  
データ漏洩 → 法令違反（GDPR） → 損害賠償（平均40億円/件）

## 対策・ベストプラクティス（500文字）  
### OWASP推奨3大防御  
#### 1. **プリペアドステートメント（必須）**  
クエリテンプレートとデータを分離：  
```java
// Java例（PreparedStatement）
String sql = "SELECT * FROM users WHERE email = ?";
PreparedStatement stmt = conn.prepareStatement(sql);
stmt.setString(1, userInput); // 自動エスケープ
```

#### 2. **ストアドプロシージャ**  
DB側でクエリを事前コンパイル：  
```sql
CREATE PROCEDURE GetUser(IN user_id VARCHAR(20))
BEGIN
  SET @sql = CONCAT('SELECT * FROM users WHERE id=', QUOTE(user_id));
  PREPARE stmt FROM @sql; EXECUTE stmt;
END
```

#### 3. **エスケープ処理（最終手段）**  
特殊文字を無害化（例：`'` → `\'`）：  
```php
// PHP例
$safe_input = mysqli_real_escape_string($conn, $input);
```

**対策比較表**:  
```
┌──────────────┬───────────┬──────────┐
│ 手法          │ 安全性    │ 実装容易性 │
├──────────────┼───────────┼──────────┤
│ プリペアド    │ ★★★★★   │ ★★★☆☆    │  
│ ストアド      │ ★★★★☆    │ ★★☆☆☆    │  
│ エスケープ    │ ★★★☆☆    │ ★★★★★    │  
└──────────────┴───────────┴──────────┘
```

> **鉄則**: 「動的クエリ生成」自体を避けるのが最善。やむを得ない場合は**ホワイトリスト検証**を追加。

## まとめ（200文字）  
SQLインジェクションは「データとコードの混同」が根源です。防御の核心は：  
1. **プリペアドステートメントの強制適用**  
2. 入力検証（長さ/文字種）の**ホワイトリスト方式実装**  
3. エラーメッセージの**情報開示制限**  
試験では「具体的な攻撃ペイロード」と「NIST/OWASPの対策番号」のセットで出題されます。次は「同一生成元ポリシー(SOP)とCSRF対策」を学び、クライアント側セキュリティを強化しましょう！  

> **最終チェック**:  
> 自社システムで`' OR 1=1 -- `を入力欄に試すのは**絶対に禁止**！ 必ず責任者とテスト環境で実施を。
