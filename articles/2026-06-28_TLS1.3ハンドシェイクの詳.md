## TLS 1.3ハンドシェイクの詳細フローと前方秘匿性：次世代暗号通信の核心  
**情報処理安全確保支援士試験 徹底解説**  

## はじめに  
TLS 1.3（Transport Layer Security 1.3）はRFC 8446で標準化された現代暗号通信の基盤です。**前方秘匿性（Forward Secrecy, PFS）** の強制実装により、過去の通信の解読リスクを根絶。試験では「プロトコルフロー」「暗号学的変更点」「実装要件」が頻出（例：令和3年午後Ⅱ問2）。本記事ではハンドシェイクフローの技術的革新とPFSの仕組みを図解します。

---

## 基本概念：TLS 1.3のコア設計思想  
### 前方秘匿性（PFS）とは  
> **「長期秘密鍵が漏洩しても、過去の通信セッションは解読不可能」** という特性。TLS 1.3ではPFSが**必須機能**に昇格（TLS 1.2ではオプション）。

```
[長期秘密鍵漏洩]  
      │  
      ▼  
┌───────────┐     ┌──────────┐  
| 過去の通信 |──×→| 解読可能 |  ✗ TLS 1.3では不可  
└───────────┘     └──────────┘  
```

### 主な変更点（TLS 1.2 vs 1.3）  
```
┌────────────────┬───────────────────────┐  
| **項目**       | **TLS 1.3**           |  
├────────────────┼───────────────────────┤  
| ハンドシェイク | 1-RTT/0-RTT           |  
| 鍵交換         | (EC)DHE必須           |  
| 暗号スイート   | AEADのみ              |  
| 廃止機能       | RSA鍵交換, SHA-1など  |  
└────────────────┴───────────────────────┘  
```

## 技術的な深堀り：ハンドシェイクフロー  
### 1-RTT基本フロー（RFC 8446 Section 2）  
```
Client　　　　　　　　　Server  
  │─ClientHello───────→│  * KeyShare, PSK拡張を含む  
  │←ServerHello───────┤  * 選択パラメータを返却  
  │←EncryptedExtensions┤  
  │←Certificate───────┤  * オプション  
  │←CertificateVerify─┤  
  │←Finished─────────→│  
  │─Finished──────────→│  
  │　　　　[Application Data]　　　│  
```

- **鍵交換の同時実行**: ClientHelloで`key_share`拡張を送信し、初回往復で鍵確立  
- **暗号スイート削減**: ネゴシエーション可能なAEAD（Authenticated Encryption）のみ  
  ``` 
  TLS_AES_256_GCM_SHA384       // 試験頻出  
  TLS_CHACHA20_POLY1305_SHA256  
  ```

### 前方秘匿性の仕組み（HKDFによる鍵導出）  
```
Client Secrets　　Server Secrets  
     │　　　　　　　　　│  
     ├─→[Early Secret]←─┤  
     │        │         │  
     ├─→[Handshake Secret]─┤  * Helloメッセージから導出  
     │        │         │  
     └─→[Master Secret]──┘  * アプリケーションデータ暗号化  
           ↓  
     [Session Key]  // セッション終了後廃棄  
```

> **HKDF（HMAC-based KDF）**: RFC 5869に基づく鍵導出。各段階で入力に**前の鍵**を使用するため、前方秘匿性が保証される。

## 攻撃・脆弱性の観点  
### ダウングレード攻撃対策  
TLS 1.3では**バージョンネゴシエーションの暗号化**を導入：
```
攻撃者（MITM）  
  │  
  │　TLS 1.2を強制  
  ▼  
┌──────┐      ┌──────┐  
|Client |──×─→|Server|  // Serverが1.3未対応と誤認させる  
└──────┘      └──────┘  
```
→ **解決策**: ServerHelloの`supported_versions`拡張で真の対応バージョンを通知（RFC 8446 Section 4.2.1）

### 0-RTTのリスク（リプレイ攻撃）  
プリシェアキー（PSK）を用いた0-RTTモードでは：
```
攻撃者  
  │  
  │ 0-RTTデータを複製  
  ▼  
┌──────┐     ┌──────┐  
|Client |     |Server|  
└──────┘     └──────┘  
```
→ **対策**: サーバー側で`replay protection`を実装（NIST SP 800-56C Rev.3）

## まとめ  
TLS 1.3は「**速度向上**」「**強制的なPFS**」「**攻撃面の縮小**」を実現した革命的なアップデートです。試験では：  
1. **1-RTTフロー**と**鍵導出プロセス**の図解を即答可能に  
2. **0-RTTのトレードオフ**（速度 vs リプレイリスク）を理解  
3. **廃止機能リスト**（RSA鍵交換, CBCモード等）を暗記  

実装時は`ssl_protocols TLSv1.3;`（nginx）や`SSLProtocol TLSv1.3`（Apache）で有効化し、常に最新の暗号スイートを採用しましょう。
