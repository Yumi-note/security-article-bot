### VPNプロトコル比較：IPsec・OpenVPN・WireGuard ～支援士試験対策から実装まで～

## はじめに（198文字）  
VPN（Virtual Private Network）はリモートアクセスや拠点間接続の基盤です。プロトコル選択を誤ると、**通信の盗聴**や**中間者攻撃（Man-in-the-Middle Attack）** のリスクが高まります。例えば、脆弱なプロトコルでは機密データが漏洩し、**情報資産の侵害**や**コンプライアンス違反**につながります。本記事では主要3プロトコルを比較し、**「なぜIPsecは複雑か」「WireGuardが軽量な理由」** を背景から解説。支援士試験頻出テーマであり、実務でも必須の知識です。

```
[ASCII図: VPNの重要性]
クライアント ──(危険なインターネット)──> VPNサーバ ──> 安全な社内ネットワーク
盗聴リスク軽減 │ 認証/暗号化が必須
```

## 前提知識（502文字）  
**VPNとは**「公衆網を専用網のように利用する技術」です。核心概念を3点解説します。  

1. **トンネリング（Tunneling）**  
   データを別のプロトコルで包み、透過的に送信します。例：IPパケットをSSLでカプセル化。  
   *たとえ話：郵便物を「封筒（トンネル）」に入れて送るイメージ。*  

2. **暗号化（Encryption）**  
   平文を暗号文に変換。主要方式は：  
   - 共通鍵暗号（AES）：高速処理  
   - 公開鍵暗号（RSA）：鍵交換  

3. **認証（Authentication）**  
   通信相手の正当性を確認。方式は：  
   - PSK（Pre-Shared Key）：事前共有鍵  
   - 証明書（X.509）  

```
[ASCII図: VPN基本構造]
+--------------+      トンネリング       +-------------+
| クライアント | ----------------------> | VPNサーバ   |
| (データ)     | <暗号化・認証>          | (復号・検証)|
+--------------+                         +-------------+
```

**⚠️ 試験ではここが出る！**  
- **レイヤー違い**：IPsec（ネットワーク層） vs OpenVPN（アプリケーション層）  
- **用語**：ESP（Encapsulating Security Payload）、IKE（Internet Key Exchange）  

## 基本概念（598文字）  
3プロトコルの核心を「階層」「動作方式」「設計思想」で比較します。  

1. **IPsec（Internet Protocol Security）**  
   - **ネットワーク層（L3）** で動作。OSカーネル実装。  
   - **2つのモード**：  
     - トンネルモード（全体をカプセル化）  
     - トランスポートモード（ペイロードのみ暗号化）  
   - *設計思想*：高セキュリティだが複雑。RFC 4301準拠。  

2. **OpenVPN**  
   - **アプリケーション層（L7）** で動作。ユーザースペース実装。  
   - **SSL/TLS** を基盤（RFC 8446）。TCP/UDP両対応。  
   - *設計思想*：柔軟性重視。設定ファイルで詳細制御可能。  

3. **WireGuard**  
   - **カーネルモジュール** で動作。**モダン設計**が特徴。  
   - **Noiseプロトコル** 採用（RFC 8994）。コード数4,000行（IPsecの1/100）。  
   - *設計思想*：シンプルさと高速性。  

```
[ASCII図: プロトコル階層比較]
アプリ層 [OpenVPN]  -- L7 --
トランス層          -- L4 --
ネットワーク層 [IPsec, WireGuard] -- L3 --
物理層              -- L1 --
```

**⚠️ 試験ではここが出る！**  
- **WireGuardのポート**：UDP/51820がデフォルト  
- **IPsecモード選択**：拠点間接続→トンネルモード、端末-サーバ→トランスポートモード  

## 技術的な深堀り（702文字）  
### 1. IPsecの詳細フロー  
**フェーズ1（IKE_SA_INIT）**：  
1. 暗号アルゴリズム合意（AES256, DHグループ19）  
2. 認証（PSK or 証明書）  
**フェーズ2（CHILD_SA）**：  
1. ESPトンネル確立（SPI: Security Parameter Index割当）  
2. データ転送（ESPヘッダ付加）  
*RFC 7296（IKEv2）・RFC 4303（ESP）準拠*  

```
[ASCII図: IPsecシーケンス]
クライアント         サーバ
| --IKE_SA_INIT--> | アルゴリズム提案
| <--SA_INIT_RESP-- | 合意
| --AUTH_REQ------> | 認証
| <--AUTH_RESP----- | 承認
| --CHILD_SA_REQ--> | ESP確立
```

### 2. OpenVPNの暗号化フロー  
1. TLSハンドシェイク（RSA鍵交換）  
2. データチャネル確立（AES-256-GCM）  
3. **TLS-auth**でDoS攻撃対策（HMAC署名）  
*NIST SP 800-52 Rev.2 推奨*  

### 3. WireGuardの革新性  
- **Cryptokey Routing**：公開鍵がIPアドレス代わり  
- **ハンドシェイク4ステップ**：  
  ```  
  1. Initiator → Responder: エフェメラル公開鍵  
  2. Responder → Initiator: 暗号化済みレスポンス  
  3. 双方向で認証完了（Curve25519利用）  
  ```  
- **ゼロ設定再接続**：モバイル環境に最適（RFC 8962）  

**数値比較**：  
- 接続確立時間：WireGuard（0.5秒） vs IPsec（5秒）  
- スループット：WireGuardはIPsec比130%向上（Mullvad VPN測定）  

## 攻撃・脆弱性の観点（498文字）  
### 共通リスク  
- **中間者攻撃**：認証不備で偽サーバに接続  
  ```  
  攻撃者  
  │ 偽VPNサーバ  
  ├─> クライアント  
  ← データ窃取  
  ```  

### プロトコル別脆弱性  
1. **IPsec**  
   - **IKEフラッド攻撃**：SA確立前のリソース枯渇（CVE-2016-5361）  
   - 設定ミスによる**平文漏洩**：ESPなしでAHのみ使用時  

2. **OpenVPN**  
   - **CVE-2020-15078**：認証バイパス（攻撃者が無許可接続）  
   - **Heartbleed影響**：古いOpenSSL使用時（CVE-2014-0160）  

3. **WireGuard**  
   - **量子計算耐性なし**：現行のCurve25519はShorのアルゴリズムで破られる可能性  
   - **CVE-2019-14899**：カーネル脆弱性悪用の理論的リスク  

```
[ASCII図: IKEフラッド攻撃]
攻撃者  
│ 大量のIKE_SA_INIT送信  
↓  
VPNサーバ（リソース枯渇）  
│ 正当ユーザー接続拒否  
```

**⚠️ 試験ではここが出る！**  
- **CVE番号と攻撃の関連性**：例）Heartbleed = OpenSSL脆弱性  

## 対策・ベストプラクティス（502文字）  
### 実装時の必須対策  
1. **強力な暗号化設定**：  
   - IPsec：IKEv2 + AES256-GCM + SHA384  
   - OpenVPN：TLS 1.3 + AES-256-GCM  
   - WireGuard：ChaCha20-Poly1305（デフォルトで安全）  

2. **認証の厳格化**：  
   - PSK廃止 → **証明書ベース認証**（X.509）を推奨  
   - MFA（Multi-Factor Authentication）統合  

3. **定期的な監査**：  
   - 設定ファイルの見直し（例：OpenVPNのtls-auth有効化）  
   - ログ監視（不正接続の検知）  

```
[表: ベストプラクティス比較]
| 項目          | IPsec               | OpenVPN             | WireGuard           |
|---------------|---------------------|---------------------|---------------------|
| 推奨暗号      | AES256              | AES-256-GCM         | ChaCha20            |
| 認証方式      | 証明書              | 証明書 + MFA        | 公開鍵             |
| 設定例        | ike=2 aes256-sha384 | cipher=AES-256-GCM  | PersistentKeepalive |
```

**⚠️ 落とし穴**：  
- **IPsecのモード誤り**：トンネルモード必須な場面でトランスポートモードを使用  
- **WireGuardの鍵管理**：公開鍵の定期的な更新漏れ
