## JWTの構造と安全な実装のポイント：支援士試験で問われる「鍵管理」と「検証漏れ」の壁を突破せよ  

⚠️ 注：本テーマは【令和4年度秋期 午後2 問2】で出題済み。認証基盤の弱点分析に必須知識！  

### はじめに：見過ごすと「乗っ取り」を許す脆弱性の巣窟  
JWT（JSON Web Token）はOAuth 2.0やOpenID Connectの基盤技術として必須ですが、誤った実装は「認証バイパス」「アカウント乗っ取り」を引き起こします。2019年のGitHubアカウント乗っ取り事件（CVE-2020-13337）でもJWT検証不備が原因でした。本記事ではRFC 7519/RFC 8725を基に「構造理解」→「攻撃パターン」→「安全実装」まで体系的に解説します。  

---

### 前提知識：トークン認証の全体像  
**JWTの役割**  
HTTPはステートレスなプロトコル（状態維持しない）のため、認証状態を維持する仕組みが必要です。従来のセッションIDはサーバー側で状態管理が必要でしたが、JWTは「クライアント側で認証情報を保持する」仕組みです。  

**関連技術の位置付け**  
```
[認証プロトコル]  
OAuth 2.0  ┬→ 認可フロー（認可コードグラントなど）  
OpenID Connect ┴→ IDトークンとしてJWTを使用  
```

**JWTの物理構造**  
3部構成の文字列（Base64Urlエンコード）で「.」で連結されます：  
```
ヘッダー.ペイロード.署名
↓ 具体例  
eyJhbGciOiJIUzI1NiIs...   # Header（アルゴリズム種別）
.eyJuYW1lIjoiV...          # Payload（ユーザー属性情報）
.SflKxwRJSMeKK...           # Signature（署名）
```

---

### 基本概念：3つのパーツで成り立つ「電子サイン付き証明書」  
#### ▶ Header（ヘッダー）：「検証ルール」の定義  
JSON形式で「署名アルゴリズム(alg)」と「トークンタイプ(typ)」を指定：  
```json
{
  "alg": "HS256",  // 署名アルゴリズム（HMAC-SHA256）
  "typ": "JWT"     // トークンタイプ
}
```

#### ▶ Payload（ペイロード）：「ユーザー属性情報」格納  
**クレーム（claim：主張）** と呼ばれる項目を格納。主に3種類：  
- **登録クレーム（Registered claims）**：予約項目（例: exp=有効期限）  
- **公開クレーム（Public claims）**：カスタム定義可能  
- **非公開クレーム（Private claims）**：当事者間合意の独自項目  

```json
{
  "sub": "user123",   // Subject（ユーザーID）
  "name": "山田太郎",
  "exp": 1717024000   // Expiration Time（Unix時間）
}
```

#### ▶ Signature（署名）：「改ざん検知」の要  
HeaderとPayloadを **Base64Urlエンコード** した文字列を結合し、指定アルゴリズムで署名生成：  
```
HMAC-SHA256(  
  base64UrlEncode(header) + "." + base64UrlEncode(payload),  
  secret_key            // 共有鍵（HS256の場合）
)
```

🔍 **たとえ話**：JWTは「**免許証**」に例えられます。  
- Header：発行機関（警察庁）と暗証番号方式  
- Payload：氏名・住所・有効期限  
- Signature：偽造防止のホログラム

### 技術的な深堀り：RFCが定める「アルゴリズム実装」の内幕  
#### JWS（JSON Web Signature）の2大署名方式  
| 方式       | アルゴリズム例 | 仕組み                     | RFC     |  
|------------|----------------|---------------------------|---------|  
| **対称鍵** | HS256/HS384    | 同一の鍵で署名/検証        | RFC7518 |  
| **非対称鍵** | RS256/ES256   | 秘密鍵で署名/公開鍵で検証  | RFC7518 |  

> ⚠️ **NIST SP 800-57推奨**：RSAは2048bit以上、ECDSAはP-256曲線必須  

#### ヘッダーのCritical処理（RFC7515 §4.1.11）  
```json
{  
  "alg":"ES256",  
  "crit":["exp"]  // 必須チェック項目を指定  
}  
```  
→ 受信側は `exp` クレームの存在を厳密に検証する義務が発生  

**署名検証フロー**  
```asciidoc
[ Client ]  
  │ 1. リクエスト（Authorization: Bearer <JWT>）  
▼  
[ Server ]  
  │ 2. "."で分割 → [Header][Payload][Signature]  
  │ 3. algヘッダーからアルゴリズム取得  
  │ 4. 鍵取得（キーID有り? ⇒ kidヘッダーで選択） 
  │ 5. 自前で署名再計算 → 受け取ったsignatureと比較  
  │ 6. 不一致 ⇒ エラー（改ざん検知）  
▼  
[DB等] expクレームの有効期限チェック  
```

### 攻撃・脆弱性の観点：なぜ「鍵管理失敗」が致命傷になるか  
#### ⚔️ 代表的な攻撃方法  
**1. アルゴリズム切り替え攻撃（Algorithm Switching）**  
```asciidoc  
攻撃者： 
  1. algヘッダーを "HS256" → "none" に改ざん  
  2. Signatureを削除  
  3. トークンを送信  
▼  
脆弱サーバー：  
  if (alg == "none") → 署名検証をスキップ ⇒ 認証バイパス成立！  
```  
→ CVE例： **CVE-2018-0114**（Cisco ASR 5000シリーズ）  

**2. キーIDインジェクション（Key ID Injection）**  
```json  
{  
  "alg": "RS256",  
  "kid": "../../../danger_key.pem"  // 悪意のパス指定  
}  
```  
→ サーバーが攻撃者の鍵を使って検証させる  

**3. 有効期限改ざん（expクレームの延長）**  
```python  
# ペイロード改ざんで永久トークン化  
import jwt  
fake_payload = { "sub":"admin", "exp": 9999999999 }  
malicious_jwt = jwt.encode(fake_payload, key=None, algorithm="none")  
```

### 対策・ベストプラクティス：現場が守る6箇条  
1. **アルゴリズムの固定検証**（ホワイトリスト方式）  
```python  
# 悪い例（任意のalgを許可）  
jwt.decode(token, key, algorithms=["HS256", "RS256"])  

# 良い例（algを明示指定）  
jwt.decode(token, key, algorithms=["RS256"])  # 固定でRS256のみ許可  
```  

2. **鍵管理の鉄則**  
- 対称鍵（HS256） ⇒ パスワード管理並の厳重保管（長さ32バイト以上）  
- 非対称鍵（RS256） ⇒ 公開鍵はjwks_uriで配信、秘密鍵はHSM（Hardware Security Module）保管  

3. **クレームの精査**  
```javascript  
// 必須チェック項目（ライブラリでの実装例）  
jwt.verify(token, key, {  
  algorithms: ["RS256"],  
  audience: "https://your-api",  // audクレーム一致確認  
  clockTolerance: 30,            // 時刻許容誤差（秒）  
  ignoreExpiration: false        // exp必須チェック  
});  
```  

4. **キーID（kid）はホワイトリスト制御**  
→ 使用許可リスト外のkidは拒否  

5. **HTTPS必須** ⇒ トークン盗聴対策（RFC8725 §6.4）  

6. **秘密鍵漏洩時は即ローテーション**（公開鍵群のバージョン管理推奨）

### 📝 試験対策ポイント：「偽装検知」問題は落とし穴だらけ  
| 論点             | 正答ポイント                     | 引っかけパターン                  |  
|------------------|---------------------------------|-----------------------------------|  
| **署名検証**     | alg指定はホワイトリスト方式のみ   | 「複数アルゴリズム対応」が偽装を許す |  
| **expクレーム**  | サーバー側時刻で評価必須          | クライアント時刻を信用すると危険    |  
| **鍵管理**       RS256は公開鍵で検証可能    | 「HS256の鍵は分散可能」は×（機密性必須） |  
| **情報漏洩**     PayloadはBase64Urlデコード可 | 「暗号化されている」と誤解（JWE≠JWS） |  

> 📌 頻出論点： **「algがnoneの場合の処理」と「kidによる外部鍵指定」は常に不正解選択肢に出現！**  

---

### まとめ：防御の根幹は「検証プロセスの完全実行」  
1. JWTは「Header.Payload.Signature」の3要素構造 ⇒ Base64Urlはエンコード方式であって暗号化ではない！  
2. アルゴリズム固定・クレーム精査・鍵管理が三大防御策  
3. 絶対に信頼するな ⇨ algヘッダー・kidヘッダー・クライアント時刻  
4. リファレンスは **RFC7519（JWT）** と **RFC8725（JWTセキュリティ）**  

次は、JWTを実際に利用する **「OAuth 2.0の認可コードフロー」** と **「OpenID ConnectのIDトークン検証」** で実践的運用を学んでください。特に「認可コード vs 暗黙的フロー」のセキュリティトレードオフは押さえましょう！

#### 署名生成フロー（RS256の場合）  
```  
[Header] -- Base64Url -->  
　　　　　　　　　　　　├→ 結合データ → [RS256署名生成] → 署名  
[Payload] - Base64Url -->　　　（秘密鍵使用）  
```  

#### JWT検証の成功条件  
```  
1. 署名の一致            ← 改ざん検知  
2. 有効期限（exp）未満   ← タイマーチェック  
3. audクレーム合致       ← トークン利用先の正当性  
4. 発行元（iss）の信用   ← 信頼できる認可サーバーか  
```
