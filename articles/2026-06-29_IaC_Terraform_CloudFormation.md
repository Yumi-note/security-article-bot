# TerraformとAWS CloudFormation：Infrastructure as Codeの基礎から実践まで

## はじめに

インフラをコードで管理する「Infrastructure as Code（IaC）」は、現代のクラウド運用に欠かせない考え方です。手作業でサーバーを構築していた時代と違い、IaCではコードを書いてバージョン管理し、何度でも同じ環境を再現できます。情報処理安全確保支援士試験でも、クラウドセキュリティやDevSecOpsの文脈でIaCの概念が問われるようになっています。

---

## 前提知識

IaCを理解するには、まず「なぜ手作業がまずいのか」を押さえることが重要です。

手作業でインフラを構築すると次の問題が起きます。

- 同じ操作を繰り返しても環境が微妙に異なる（スノーフレーク問題）
- 誰が何を変更したか記録が残らない
- 本番・ステージング・開発環境で設定が少しずつズレる

IaCはこれを解決します。インフラの構成を宣言的なコード（HCLやYAML）で記述し、ツールが自動で環境を構築・変更・削除します。

```
【従来の手作業】
開発者 → AWSコンソール → ポチポチ → 環境構築（再現性なし）

【IaCの場合】
開発者 → コード記述 → ツール実行 → 環境構築（何度でも同じ結果）
```

関連する重要用語を整理します。

| 用語 | 意味 |
|------|------|
| 宣言的記述 | 「何を作るか」を書く（「どう作るか」は書かない） |
| 冪等性（べきとうせい） | 何度実行しても同じ結果になる性質 |
| ステート（状態管理） | 現在のインフラ状態をファイルで追跡 |
| プロビジョニング | リソースを作成・設定する操作 |

## 基本概念：TerraformとCloudFormationの違い

2つのツールは目的は同じですが、設計思想が異なります。

```
┌─────────────────────────────────────────────────┐
│              IaCツール比較                        │
├──────────────┬──────────────┬───────────────────┤
│  項目        │  Terraform   │  CloudFormation   │
├──────────────┼──────────────┼───────────────────┤
│  開発元      │  HashiCorp   │  AWS              │
│  対応クラウド│  マルチ対応  │  AWSのみ          │
│  記述言語    │  HCL / JSON  │  YAML / JSON      │
│  料金        │  OSS版無料   │  完全無料         │
│  状態管理    │  tfstateファイル│ AWSが管理       │
│  学習コスト  │  中程度      │  低め（AWS前提）  │
└──────────────┴──────────────┴───────────────────┘
```

Terraformは「どのクラウドでも同じ書き方で書ける」汎用ツール。CloudFormationは「AWS専用だが、AWSとの統合が深く、追加費用ゼロ」という位置づけです。

## 技術的な深堀り

### Terraformのしくみ

Terraformは3つのフェーズで動作します。

```
① terraform init
   ↓ プロバイダー（AWSやGCPのプラグイン）をダウンロード

② terraform plan
   ↓ 現在の状態（tfstate）と理想の状態（コード）を比較
   ↓ 「何を追加・変更・削除するか」をプレビュー表示

③ terraform apply
   ↓ 実際にリソースを作成・変更・削除
   ↓ 完了後にtfstateを更新
```

Terraformのコード例（EC2インスタンスを1台作る場合）：

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t3.micro"

  tags = {
    Name = "WebServer"
  }
}
```

⚠️ 試験ではここが出る！`terraform plan` は変更内容を確認するだけで、実際には何も変更しません。本番環境では必ずplanで確認してからapplyします。

### CloudFormationのしくみ

CloudFormationは「スタック」という単位でリソースをグループ管理します。

```
CloudFormationテンプレート（YAML）
        ↓
   AWSに送信
        ↓
  スタック作成
   ├── EC2インスタンス
   ├── セキュリティグループ
   ├── S3バケット
   └── RDSインスタンス
        ↓
  スタック削除 → 関連リソースを一括削除
```

CloudFormationのコード例（同じくEC2を1台）：

```yaml
Resources:
  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: ami-0abcdef1234567890
      InstanceType: t3.micro
      Tags:
        - Key: Name
          Value: WebServer
```

## 攻撃・脆弱性の観点

IaCはセキュリティリスクも抱えています。

```
【IaCのリスクフロー】

攻撃者がリポジトリに侵入
        ↓
IaCコードに悪意ある変更を追加
（例：セキュリティグループを全開放、
      バックドアユーザーを追加）
        ↓
CI/CDパイプラインが自動でapply
        ↓
本番環境に脆弱な設定が展開される
```

主な脆弱性パターンを整理します。

| リスク | 内容 | 具体例 |
|--------|------|--------|
| シークレットの埋め込み | APIキーをコードに直書き | `access_key = "AKIA..."` |
| 過剰な権限 | IAMロールに `*` 権限を付与 | `Action: "*"` |
| 暗号化の欠如 | S3バケットの暗号化無効 | `encryption = false` |
| 公開状態の誤設定 | S3バケットをPublicに | `acl = "public-read"` |

2019年に発覚したCapital Oneのデータ漏洩事件は、IaCで作成されたEC2インスタンスのIMDS（インスタンスメタデータサービス）がSSRF脆弱性と組み合わさった事例として有名です。

## 対策・ベストプラクティス

```
【安全なIaC運用の構成】

コード変更
  ↓
静的解析（tfsec / cfn-nag）← セキュリティチェック
  ↓
プルリクエスト + レビュー
  ↓
terraform plan の結果を確認
  ↓
承認後にapply（本番は手動承認必須）
  ↓
変更ログを記録
```

具体的な対策です。

- シークレットはコードに書かず AWS Secrets Manager / HashiCorp Vault を使う
- `tfsec`（Terraform）や `cfn-nag`（CloudFormation）で静的解析を自動化
- IAMロールは最小権限の原則に従い、`*` を避ける
- S3バケットは `block_public_acls = true` を必ず設定
- tfstateファイルは S3 + DynamoDB で暗号化・ロック管理する

## 📝 試験対策ポイント

| 出題テーマ | 覚えるポイント |
|-----------|---------------|
| IaCの目的 | 再現性・冪等性・バージョン管理 |
| terraform plan | 変更のプレビュー。実変更はapply |
| tfstateの役割 | 現在のインフラ状態を記録するファイル |
| 最小権限の原則 | IaCでもIAMは必要最低限の権限のみ |
| DevSecOps | セキュリティをCI/CDパイプラインに組み込む |

⚠️ 試験ではここが出る！「宣言的」と「手続き的」の違いを押さえておきましょう。TerraformもCloudFormationも宣言的（何を作るかを記述）であり、Ansibleなど手続き的ツールとは設計思想が異なります。

---

## まとめ

TerraformとCloudFormationはどちらもIaCの代表ツールですが、マルチクラウド対応か・AWS専用かで使い分けます。セキュリティ観点では、コードにシークレットを埋め込まない・最小権限を徹底する・静的解析をCI/CDに組み込むことが重要です。次は「DevSecOpsとシフトレフトセキュリティ」でパイプライン全体のセキュリティを学ぶと理解が深まります。
