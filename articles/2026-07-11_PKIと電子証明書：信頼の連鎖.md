# PKIと電子証明書：信頼の連鎖とX.509の仕組み

## はじめに

情報処理安全確保支援士の取得を目指すのであれば、PKI（Public Key Infrastructure）と電子証明書は必ず理解しなければならない基本的なセキュリティ技術です。PKIは公開鍵暗号技術を基盤とし、信頼できる証明書をissued（発行）するために役割を果たします。この構造を理解することで、安全な通信や認証プロトコルがどのように機能するかが明確になります。

しかし、多くのITエンジニアにとって、PKIと電子証明書は未知の領域です。ここでは、初学者から中級者の理解を深めるために、PKIと電子証明書の基本から技術的な詳細まで、わかりやすく解説します。

## 前提知識

**公開鍵暗号**

公開鍵暗号は暗号化と復号化に用いられる鍵が異なる暗号方式です。公開鍵（公開鍵を持つ者がだれでもデータを暗号化できる）と秘密鍵（公開鍵の持ち主だけがこれで復号化可能）という形となります。一般的なアルゴリズムとしてはRSAが広く使われます。

**CA（Certification Authority）とRA（Registration Authority）**

CAは電子証明書を発行する組織のことであり、CA自身の公開鍵と電子証明書が連動しています。RAはCAに申請された情報を検証する役割を持ちます。

**信頼の連鎖（Trust Chain）**

信頼の連鎖とは、CAが発行した電子証明書が本物であることを確認する仕組みで、自証明書にたどり着くまでに作られる連鎖です。

```
+---------------+
|  Root CA   |
+---------------+
       |
       |
       v
+---------------+
|  中間CA  |
+---------------+
       |
       |
       v
+---------------+
|  サーバ証明書 |
+---------------+
```

## 基本概念

PKIでは、証明書を発行するCA、認証するRA、中間CAが複数階層になっていることで信頼関係が構築されます。これは信頼の連鎖を形成するために重要です。

- **Root CA：** 最上位に位置するCAで、自己署名証明書を持ちます。Root CAを信頼することがPKIの出発点となります。
- **中間CA：** Root CAより下のCAで、上位CAより発行された証明書を持ちます。最終的なサーバ証明書やクライアント証明書の発行に利用されます。
- **サーバ証明書・クライアント証明書：** 中間CAより発行される、サーバやクライアントを特定するための電子証明書。HTTPS通信などにおけるサーバの認証や、サーバからのクライアント認証に利用されます。

## 技術的な深堀り

X.509はPKIにおける電子証明書の標準となる規格であり、RFC 5280で定義されています。ここでは、X.509証明書の構造と証明書の検証プロセスの技術的な詳細について解説します。

- **X.509の構造：** X.509証明書はTBS (To Be Signed) Certi- ficate、Algorithm Identifier、Signature Valueから構成されます。TBS Certificateには発行者情報、有効期間、発行対象情報などが含まれます。
- **署名アルゴリズム：** X.509証明書の署名は通常、RSAやECDSAなどの公開鍵暗号アルゴリズムによって行われます。

```
+-----------------------------------------------+
|  Certificicate  |
|                  |
|  +------------+  |
|  |  TBS      |  |
|  +------------+  |
|                  |
|  +------------+  |
|  | Algorithm  |  |
|  +------------+  |
|                  |
|  +------------+  |
|  |  Signature  |  |
|  +------------+  |
+-----------------------------------------------+
```

- **証明書検証プロセス：** 
  1. **証明書の取得**：クライアントはサーバからサーバ証明書を受け取ります。
  2. **発行者のCAの取得**：サーバ証明書から発行者のCA情報を取得します。
  3. **発行者のCAの証明書の取得**：発行者のCAの証明書を取得します。これは通常中間CAです。
  4. **Root CAの取得**：中間CA証明書をたどってRoot CA証明書を取得します。
  5. **署名検証**：各証明書の署名を検証します。

## 攻撃・脆弱性の観点

PKIの脆弱性や、実際に発生している攻撃についてみていきましょう。

- **Heartbleed脆弱性**：あるTLSで利用されているHeartbeat拡張を攻撃することで、接続先サーバのメモリ内容（秘密鍵の取得）の一部が取得できる脆弱性。2014年に発見され、広範囲のサービスで被害が報告されました。
- **特権エスカレーション攻撃**：管理者権限を取得することで、Root CAなどの発行に使用されている秘密鍵にアクセスし、任意の証明書を発行できる脆弱性。

```
+-----------+
|  攻撃者  |
+-----------+
       |
       |
       v
+-----------+
| サーバに  |
| 接続する  |
+-----------+
       |
       |
       v
+-----------+
| Heartbeat |
| を送信   |
+-----------+
       |
       |
       v
+-----------+
| メモリから  |
| 秘密鍵を取得 |
+-----------+
```

## 対策・ベストプラクティス

- **証明書の適切な期限設定**：証明書の有効期限は必要最小限に設定しましょう。
- **証明書の適切な権限設定**：証明書に関連した秘密鍵などの権限設定は限られた機密性を持つべきです。
- **CAを適切に選択する**：信頼できるCAを選択することで、信頼の連鎖の強度が保証されます。
- **認証アルゴリズムの利用**：RSA-2048やECDSAなどの強力なアルゴリズムを利用しましょう。

| 対策 | ベストプラクティス |
| :---- | :---------------- |
| 証明書の期限設定 | 最小限の期限設定 |
| 証明書の権限設定 | 限られた権限設定 |
| CAの選択 | 信頼できるCA |
| 認証アルゴリズム | RSA-2048やECDSA |
