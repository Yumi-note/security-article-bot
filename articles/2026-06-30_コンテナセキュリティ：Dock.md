## コンテナセキュリティの要：Dockerの「隔離」は本当に安全か？ Namespaces/cgroupsの仕組みと脆弱性対策

![コンテナセキュリティイメージ図](https://example.com/container-security.png)  
*図1: コンテナの隔離イメージ*

## はじめに
コンテナ技術は開発効率を飛躍的に向上させましたが、その軽量さゆえにセキュリティを過信する傾向があります。特にDocker環境では、デフォルト設定のまま運用すると「コンテナブレイクアウト」と呼ばれる深刻な攻撃を受け、ホストOS全体を乗っ取られるリスクがあります。実際、2022年にはDocker Engineの脆弱性(CVE-2022-24730)が発見され、深刻度「高」と評価されました。本記事では、Dockerの隔離機構の核心である**Linuxカーネル機能**の仕組みを解き明かし、その脆弱性と対策を解説します。

## 前提知識
### コンテナの本質
コンテナは「プロセスの隔離環境」です。仮想マシン(VM)のようにハードウェアをエミュレートせず、ホストOSのカーネルを共有します。これにより高速起動・低リソース消費を実現します。

### キーワード解説
- **Linuxカーネル(Linux Kernel)**: OSの中核。リソース管理やプロセス制御を担当
- **Namespaces(名前空間)**: プロセスが参照可能なリソース（PID, ネットワーク等）を分離する仕組み
- **cgroups(Control groups)**: CPU/メモリ/ディスクI/Oなどのリソース使用量を制限・管理
- **コンテナランタイム(Container Runtime)**: コンテナの実行を管理するソフトウェア（例: containerd, runc）

```plaintext
[ホストOSカーネル]
    |
    ├── [Namespace A]  # コンテナAの隔離環境
    │   ├── プロセス1
    │   └── プロセス2
    │
    └── [Namespace B]  # コンテナBの隔離環境
        ├── プロセス1
        └── プロセス3
```
*図2: カーネル共有とNamespacesの概念図*

## 基本概念
### Dockerの隔離を支える2大機構
#### 1. Namespacesによるリソース隔離
| Namespaceタイプ | 隔離対象           | デフォルト有効 |
|-----------------|--------------------|----------------|
| PID             | プロセスID         | ○             |
| Network         | ネットワークスタック | ○             |
| Mount           | ファイルシステム    | ○             |
| UTS             | ホスト名・ドメイン名| ○             |
| IPC             | プロセス間通信      | ○             |
| User            | ユーザー/グループID | ✕ (要設定)    |

#### 2. cgroupsによるリソース制限
```plaintext
[ホストOS]
├── cgroup (CPU制限) 
│   └── docker/コンテナID
│       ├── cpu.max   # CPU使用上限
│       └── cpu.stat  # 使用状況
│
└── cgroup (メモリ制限)
    └── docker/コンテナID
        ├── memory.max      # メモリ上限
        └── memory.current  # 現在使用量
```
*図3: cgroupsによるリソース制限の構造*

⚠️ **試験で注意**: Namespacesは「可視性」の隔離であって「アクセス権」の制限ではない。ホストのリソースにアクセス可能な場合、突破される危険性がある。

## 技術的な深堀り
### Namespacesの実装メカニズム
NamespacesはLinuxカーネルのシステムコールで実装されます。主要なシステムコールとRFC:

- `clone()`: 新しいNamespaceでプロセス生成 (Linux man-pages)
- `unshare()`: 実行中のプロセスをNamespaceから分離
- `setns()`: 既存のNamespaceにプロセスを参加

**重要仕様**: Linuxカーネルドキュメント (Documentation/namespaces) で定義。特にUser NamespaceはUID/GIDをホストと独立させますが、デフォルトでは無効（`docker run --userns=host`がデフォルト）。

### cgroupsのバージョンと制御方法
- **cgroups v1**: 階層型構造（従来型）
- **cgroups v2**: 統一階層モデル（Linux 4.5以降）
  - メモリ保護機能強化（`memory.high`によるソフト制限）
  - NIST SP 800-190 (コンテナセキュリティガイド) で必須推奨事項

```plaintext
# cgroups v2でのCPU制限設定例
$ docker run -it --cpus="0.5" ubuntu /bin/bash

# 対応するシステムファイル
/sys/fs/cgroup/docker/<container-id>/cpu.max
→ "50000 100000" # 50%利用制限
```

## 攻撃・脆弱性の観点
### コンテナブレイクアウト攻撃フロー
```plaintext
攻撃者 → 脆弱なアプリ(コンテナ内) 
        → コンテナ内root権限取得 
        → Namespaceの設定不備を悪用
        → ホストOSの/procや/sysにアクセス
        → runcの脆弱性(CVE-2019-5736)等を利用
        → ホストOSのrootシェル獲得
```
*図4: コンテナブレイクアウトの攻撃経路*

### 代表的な脆弱性事例
1. **CVE-2019-5736 (runcの脆弱性)**
   - 深刻度: Critical (CVSS 7.2)
   - 影響: コンテナ内からホストのruncバイナリ上書き可能

2. **CVE-2022-24730 (Docker Engine認証問題)**
   - 深刻度: High (CVSS 7.8)
   - 影響: 認証なしでDocker APIにアクセス可能に

⚠️ **試験必須知識**: PID=1のプロセスが特権プロセスになる点を悪用した攻撃（例: --privilegedフラグの誤用）

## 対策・ベストプラクティス
### 多層防御の実装
```plaintext
[対策レイヤー]
1. コンテナアプリ層: アプリ脆弱性対策
2. ランタイム層: セキュアな設定
3. ホストOS層: カーネルハードニング
4. オーケストレーション層: ネットワークポリシー
```
*図5: 多層防御モデル*

### 必須設定項目
1. **非特権ユーザー実行**
   ```bash
   docker run --user 1000:1000 my-app # 非rootユーザー指定
   ```

2. **Namespaceの強化**
   ```bash
   docker run --pid=host --userns=host # 危険！隔離が無効化
   docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE # 権限最小化
   ```

3. **cgroupsによる制限**
   ```bash
   docker run --memory="500m" --cpus="1.0" --blkio-weight=500
   ```

4. **Rootlessモードの採用**
   ```bash
   # Dockerデーモンをroot権限で動かさない
   dockerd-rootless-setuptool.sh install
   ```

## 📝 試験対策ポイント
| 出題ポイント              | 正答例                     | 誤答例                     |
|--------------------------|----------------------------|----------------------------|
| Namespacesの役割         | リソースの可視性を分離     | 物理リソースを完全に分離   |
| cgroupsの機能            | リソース使用量の制限       | プロセスの完全な分離       |
| コンテナブレイクアウト   | User Namespace無効が原因   | 仮想化の欠陥              |
| 安全なデフォルト設定      | `--userns=remap`の有効化   | `--privileged`の使用      |
| セキュアなイメージ構築   | マルチステージビルドの利用 | 全ツールを含む巨大イメージ |

⚠️ **引っかけ問題パターン**: 「Dockerは完全な仮想化技術である」→ ×（共有カーネルモデル）

## まとめ
Dockerの隔離機構はNamespaces/cgroupsというカーネル機能に依存しており、その設定不備が重大な脆弱性を生みます。特に「デフォルト設定の危険性」「User Namespaceの無効化」「過剰な権限付与」は現場でも試験でも最重要ポイントです。次に学ぶべきは「Kubernetesセキュリティ」です。NetworkPoliciesやPodSecurityPolicy(PSP)/PodSecurity Admission(PSA)でクラスタレベル防御を強化しましょう。

> コンテナは「軽量な監獄」に過ぎない。鍵（設定）がかかっていなければ、脱走は時間の問題だ。セキュリティはデフォルトでは存在せず、設計によって築かれる。
